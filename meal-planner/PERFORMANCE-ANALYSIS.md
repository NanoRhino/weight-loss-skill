# 7日饮食计划生成性能分析

## 概述

7日饮食计划（Step 4）是整个 meal-planner skill 中最耗时的环节。SKILL.md 本身已承认这一点，要求在生成前向用户发送"大约需要 1–2 分钟"的等待提示。本文档全面分析造成速度慢的根本原因。

---

## 瓶颈 1：单次 LLM 输出的 Token 量极大

**这是最核心的瓶颈。**

7日饮食计划要求生成一个完整的 HTML 文件，包含：

| 组成部分 | 估算内容量 |
|---|---|
| HTML/CSS 样板（`<style>` 块 + 结构标签） | ~300 行 CSS + 页面骨架 |
| Summary Card | 1 个摘要卡片 |
| 7 个 Day Card × 每天 4 个 Meal Block | 28 个餐食块 |
| 每个 Meal Block 含：标题 + 宏量 + dish-summary + 3-5 个 food-list `<li>` + 可选 meal-tip | 每个 ~8-12 行 HTML |
| Footer | 1 个页脚 |
| **总计** | **约 500-800 行 HTML，估计 8,000-15,000 tokens** |

LLM 的输出速度受限于逐 token 生成的本质。一个完整的 7天计划需要输出的 token 数量相当于一篇中型文章，这是速度慢的第一性原理原因。

**对比：** Diet Template（Step 3）只需要生成 1 天的模板 + 示例，约 30-50 行文本，生成时间在 10-20 秒内。7天计划的内容量是其 **10-20 倍**。

---

## 瓶颈 2：每一餐都需要满足多维约束的推理

生成不是简单的模板填充。对于 7天 × 4餐 = 28 个餐食，LLM 在生成每一餐时都需要同时满足：

### 营养约束
- 每餐的热量需要匹配分配比例（早 25% / 午 30% / 晚 30% / 零食 15%）
- 每餐的 P/C/F 三大宏量需要各自正确
- 每天的总热量和宏量需要加总到日目标范围内（±100 kcal）
- 蛋白质来源于体重计算（weight_kg × 1.2-1.6g），脂肪按饮食模式百分比，碳水填充剩余

### 食物多样性约束
- 7天内午餐和晚餐的主菜不能重复（Step 6 规则 4）
- 蛋白质来源需要轮换（鸡胸肉、鱼、豆腐、牛肉等）
- 烹饪方式需要变化（煎、烤、炖、蒸等）
- 早餐允许少量重复，但午晚餐每天必须不同

### 储存可行性约束（meal-prep-feasibility.md）
- Tier A 食物（炖肉、咖喱等）才能用于批量准备日
- Tier C 食物（叶菜炒制、油炸食品、面条汤等）只能安排在现做日或外食日
- 同一批量准备的菜最多连续出现 3 天
- 鱼类最多储存 1-2 天，不能作为批量准备蛋白质
- 不同储存特性的组件必须分开（酱汁和面条、汤和粉丝等）

### 鸡蛋限制
- 每天最多 1 个全蛋
- 如果早餐已含蛋，午晚餐不能安排以蛋为主的菜（番茄炒蛋、蛋花汤等）

### 份量精度约束
- 最小粒度 0.5（不能出现 0.3、0.7）
- 可数食物必须用整数（1 个蛋、2 片面包，不能 0.5 个蛋）
- 每个食物需要自然描述 + 精确重量两种表达

### 本地化约束
- 食物名称、计量单位、餐食结构都需要适配用户的地区
- 中国用户：米饭/粥/豆浆/豆腐，克为单位
- 美国用户：燕麦/酸奶/三明治，oz/cup 为单位

**每一餐的生成都不是独立的——它需要 "回看" 已生成的内容来确保不重复、不超标。** 这种全局一致性检查是 LLM 推理的重大负担。

---

## 瓶颈 3：HTML 模板的严格结构要求

SKILL.md Step 6 明确要求：

> **CRITICAL: The HTML meal plan MUST follow the structure defined in `templates/meal-plan.html`. Do not deviate from the CSS classes, nesting, or element hierarchy.**

这意味着 LLM 不能用简化格式输出再后处理，必须直接生成符合以下层级的完整 HTML：

```
.day-card
  .day-header (h2 + .day-macros)
  .day-body
    .meal-block (×4)
      .meal-title (h3 + .meal-macros)
      .dish-summary
      .food-list (li × 3-5, 每个 li 含 .portion span)
      .meal-tip (可选)
```

HTML 标签本身就是大量冗余 token（`<div class="meal-block">`, `<span class="portion">`, `</li>` 等）。结构性标记可能占输出 token 的 30-40%，这些 token 不携带营养信息但必须正确生成。

---

## 瓶颈 4：全量 CSS 内联要求

模板的 `<style>` 块有 ~300 行 CSS（包含 Reset、Page、Header、Summary Card、Day Card、Meal Block、Eating-out、Grocery、Footer、Responsive、Print 等所有样式）。

SKILL.md 要求：
> Keep **all CSS inline** in a `<style>` block — the file must be fully self-contained with no external dependencies.

LLM 需要在输出的开头生成完整的 CSS 块（约 1,500-2,000 tokens），然后才能开始生成实际内容。这增加了首次 token 到有意义内容之间的延迟。

---

## 瓶颈 5：外食/堂食餐的额外复杂性

当用户有外食需求时，eating-out 类型的 meal-block 需要额外生成：
- `<span class="tag">` 标签（外卖/堂食/便利店）
- `.order-info` 具体餐厅和菜品
- 点餐指导（具体怎么点、加什么减什么）
- `.meal-tip` 实用建议

这比纯家庭烹饪餐的生成复杂度更高，且需要具备特定餐厅/外卖平台的知识。

---

## 瓶颈 6：没有预计算或缓存机制

当前架构是**纯 LLM 单次生成**，没有任何中间步骤：

```
用户确认 → LLM 读取约束 → LLM 一次性生成完整 HTML → Write 工具写入文件
```

对比可能的优化架构：

```
用户确认 → planner-calc.py 计算宏量分配 → 生成7天菜单骨架 → 逐天填充细节 → 组装 HTML
```

`planner-calc.py` 只提供宏量计算（`macro-targets` 命令），不提供：
- 每天每餐的具体热量/宏量分配
- 食物数据库查询
- 菜品自动搭配
- HTML 模板自动填充

**所有这些都由 LLM 在一次推理中完成**，没有任何工具辅助来分担计算负载。

---

## 瓶颈 7：输入上下文窗口的压力

生成 7天计划时，LLM 的输入上下文中需要同时加载：

| 上下文来源 | 估算 token 数 |
|---|---|
| SKILL.md（完整指令） | ~4,000-5,000 |
| meal-prep-feasibility.md | ~2,000 |
| meal-plan.html（模板参考） | ~1,500 |
| 用户对话历史（Step 1-3 的交互） | ~1,000-2,000 |
| health-profile.md / health-preferences.md | ~500-1,000 |
| PLAN.md（减重计划数据） | ~500-1,000 |
| **总输入上下文** | **~10,000-15,000 tokens** |

加上输出的 ~10,000-15,000 tokens，一次 7天计划生成的总 token 消耗可能达到 **20,000-30,000 tokens**。更大的上下文窗口使用量会直接影响推理速度。

---

## 定量估算

| 指标 | 估算值 |
|---|---|
| 输出 token 数 | 8,000-15,000 |
| LLM 输出速率（典型） | ~50-80 tokens/秒 |
| 纯输出时间 | 100-300 秒（1.5-5 分钟） |
| 输入处理时间 | ~5-15 秒 |
| 工具调用开销（Write 文件） | ~1-3 秒 |
| **总预估时间** | **~2-5 分钟** |

SKILL.md 声称"1-2 分钟"可能偏乐观，实际生成时间可能更接近 2-4 分钟。

---

## 优化建议

### 短期（不改架构）

1. **CSS 外部化/预存** — 将 300 行 CSS 存为静态文件，LLM 只需生成 HTML body 内容。用简单的 shell 脚本拼接 head + body + footer，可节省 ~1,500-2,000 输出 tokens（~15-20% 输出量）。

2. **分天生成** — 不要求一次性输出全部 7天，改为逐天生成然后拼接。这样每次生成的约束检查范围更小（只需回看已生成的 2-3 天），但总时间可能相近（多次 API 调用有额外延迟）。

3. **更精确的等待时间提示** — 将"1-2 分钟"改为"2-4 分钟"以匹配实际体验。

### 中期（轻度架构改动）

4. **引入食物数据库脚本** — 创建 `meal-calc.py` 工具脚本，包含常见食物的热量/宏量数据库。LLM 决定菜品组合，脚本验证并调整份量。这可以减少 LLM 在数值计算上的推理负担。

5. **模板化 HTML 生成** — 使用 Python 模板引擎（Jinja2），LLM 只需输出结构化 JSON（菜品列表 + 份量），由脚本渲染为 HTML。输出 token 量可降低 50-60%。

6. **批量准备日程预计算** — 将 2-3-2 模式（周日批量→周一到三，周三批量→周四五）的日程分配从 LLM 推理中抽离，由脚本根据用户的烹饪条件自动生成骨架。

### 长期（架构重设计）

7. **LLM 输出 JSON + 脚本渲染 HTML** — 彻底分离内容生成和格式渲染。LLM 只输出结构化数据：
   ```json
   {
     "days": [
       {
         "meals": [
           {
             "type": "breakfast",
             "dishes": ["燕麦粥", "水煮蛋", "牛奶"],
             "foods": [
               {"name": "燕麦", "portion": "半碗", "weight_g": 40, "cal": 150, "p": 5, "c": 27, "f": 3}
             ]
           }
         ]
       }
     ]
   }
   ```
   JSON 输出比 HTML 节省约 **60-70% 的 token 量**，同时更容易进行后续的宏量验证和一致性检查。

8. **并行生成** — 将 7天拆分为多个独立的 LLM 调用（如 Day 1-2, Day 3-4, Day 5-6, Day 7），并行执行。需要预先规划好菜品不重复的约束，但可以将生成时间缩短到接近单天生成的时间。

---

## 根因总结

| 排名 | 根因 | 影响程度 | 可优化性 |
|---|---|---|---|
| 1 | 输出 token 量巨大（完整 HTML × 7天 × 4餐） | **极高** | 中期可通过 JSON+模板 大幅改善 |
| 2 | 每餐需满足多维约束的串行推理 | **高** | 长期可通过工具辅助分担 |
| 3 | HTML 结构标签的冗余 token 开销 | **中** | 短期可通过 CSS 外部化部分改善 |
| 4 | 无预计算/缓存，全靠单次 LLM 推理 | **中** | 中期可通过食物数据库脚本改善 |
| 5 | 大输入上下文压力 | **低-中** | 可通过精简指令或分步执行改善 |
| 6 | 外食餐的额外生成复杂性 | **低** | 本身合理，影响较小 |

**一句话总结：7天饮食计划慢的本质原因是要求 LLM 在单次推理中同时扮演营养师（多维约束推理）+ 前端开发（HTML 渲染）+ 数据库（食物营养数据），输出量达到 10,000+ tokens，且没有任何工具辅助来分担这些职责。**
